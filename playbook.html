Covers pre-encryption detection, containment, negotiation decision gates, recovery sequencing, and post-incident hardening. Reference cases: LockBit 3.0, BlackCat/ALPHV, Cl0p.

CRITICAL: Do NOT reboot any affected system — you will destroy volatile memory artifacts including in-memory encryption keys, injected shellcode, and active C2 sessions. Capture RAM before any power change.
01 Detection & Initial Validation T+0 → T+15min
  • 1.1
    Confirm indicator type — encryption in progress vs already complete?
    EDR alert, ransom note, renamed files (*.locked, *.{uuid}), shadow copy deletion events (EventID 524 / vssadmin logs)
    2min
  • 1.2
    Identify affected system count via EDR telemetry
    Query EDR for hosts with shadow copy deletion, large-scale file rename, vssadmin.exe delete shadows, or wmic shadowcopy delete in last 60 minutes
    5min
  • 1.3
    Verify encryption is active or stopped — do NOT reimage without imaging first
    If encryption is still running: isolate network first, then image RAM. If already stopped: image RAM, then disk
    3min
  • 1.4
    Escalate: notify IR lead, CISO, Legal, and PR immediately
    Start the incident timeline document. Timestamp every action from this point. Log all named responders and their roles
    5min
02 Forensic Acquisition & Triage T+15 → T+45min
  • 2.1
    Acquire RAM from all affected systems before any power change
    Use winpmem, DumpIt, or FTK Imager. Target: patient-zero and any system showing active encryption. RAM may contain keys, injected code, or C2 strings
    20min
  • 2.2
    Image disk(s) of patient-zero system
    dd if=/dev/sda bs=4M | gzip > evidence.img.gz or use FTK Imager (Windows). Verify SHA-256 of image immediately
    variable
  • 2.3
    Collect volatile network state on affected systems
    netstat -ano > netstat.txt · ipconfig /all > ipconfig.txt · arp -a > arp.txt · tasklist /v > tasklist.txt
    3min
  • 2.4
    Collect Windows Event Logs before isolation
    Export Security (4624/4625/4648/7045/1102), System, Application, PowerShell/4104 scriptblock logs
    5min
PS> winpmem_mini_x64_rc2.exe --output memdump.aff4 --format aff4 PS> Get-EventLog -LogName Security -Newest 5000 | Export-Csv security.csv PS> wevtutil epl "Microsoft-Windows-PowerShell/Operational" ps-scriptblock.evtx
03 Containment T+45 → T+2h
Sequencing matters: Acquire evidence BEFORE isolating. Isolation ends your visibility window — once a host is network-isolated, you lose live C2 telemetry and the ability to pivot to new hosts from that socket.
  • 3.1
    Isolate affected hosts at network layer (VLAN isolation, not power-off)
    EDR agent network isolation, or switch VLAN reassignment. Document which hosts are isolated and when
    15min
  • 3.2
    Block C2 indicators at perimeter — only after scoping is complete
    Block identified IPs, domains, and JA3 hashes at firewall/proxy. Sinkhole DGA domains if still beaconing (preserve logs before blocking)
    10min
  • 3.3
    Disable compromised accounts (especially service and admin accounts)
    Check for pass-the-hash / Kerberos ticket reuse. Disable accounts seen in malicious authentication events (4648, 4768, 4769)
    20min
  • 3.4
    Determine exfiltration scope BEFORE negotiation — critical gate
    Review proxy/DLP/firewall logs for large outbound transfers 24–72h before encryption. LockBit/ALPHV exfil before encrypting. Ransom demand doubles if you admit exfil in negotiation without knowing scope
    30min
04 Investigation & Root Cause T+2h → T+24h
  • 4.1
    Identify patient-zero and initial access vector
    Analyze: phishing email (MX/mail server logs), VPN/RDP access (4624/4625 with logon type 10), exploited public-facing service (web logs). Trace EVTX timestamps backward from first encrypted file
    2h
  • 4.2
    Map lateral movement path
    SMB (EventID 5140/5145), WMI (EventID 4688 with wmic/wmiprvse), PsExec (EventID 7045 PSEXESVC), RDP (4624 logon type 10), Kerberoasting (4769 RC4 tickets)
    3h
  • 4.3
    Analyze malware sample — identify family and encryptor version
    Submit to sandbox (Any.run, Hybrid Analysis). Run YARA against LockBit/ALPHV/Cl0p rule sets. Identify encryptor (RSA-wrapped AES? ChaCha20? Salsa20?). Family ID determines decryption viability
    2h
  • 4.4
    Run Volatility against RAM captures — validate process injection, keys
    vol3 -f mem.dmp windows.malfind · vol3 -f mem.dmp windows.netscan · vol3 -f mem.dmp windows.cmdline. Some encryptors leave key material in process memory if encryption didn't complete
    2h
# Volatility 3 — ransomware investigation commands $ python3 vol.py -f hermes07.raw windows.pslist | grep -v Idle $ python3 vol.py -f hermes07.raw windows.malfind --dump-dir ./dumps/ $ python3 vol.py -f hermes07.raw windows.netscan | grep -v CLOSE $ python3 vol.py -f hermes07.raw windows.cmdline $ python3 vol.py -f hermes07.raw windows.dlllist --pid 4812 # Identify ransom family via YARA $ yara -r lockbit3-rules.yar /mnt/evidence/ $ yara -r alphv-blackcat.yar /mnt/evidence/ # Shadow copy check PS> vssadmin list shadows PS> Get-WinEvent -FilterHashtable @{LogName='System';Id=7045} | Where-Object {$_.Message -match 'vsserv|vss'}
GATE Negotiation Decision

Do NOT negotiate if:

Working backups are confirmed clean (pre-infection) · Decryptor is publicly available (Europol No More Ransom) · Legal/jurisdiction prohibits payment to sanctioned entity · Data exfil scope is limited or confidential data is not confirmed exfiltrated

Consider negotiation only if:

Backups are also encrypted or confirmed compromised · Critical systems (hospital, OT) will cost lives · Confirmed exfil of sensitive PII/trade secrets and negotiation may reduce public disclosure · Legal has cleared the entity (not on OFAC SDN list)

NEVER pay before: Confirming entity is not OFAC-sanctioned (criminal penalty) · Getting Legal sign-off · Verifying decryptor actually works (test one non-critical file) · Securing cyber insurance notification
05 Recovery Sequencing T+24h → T+120h
  • 5.1
    Rebuild from known-good backups in isolated environment first
    Confirm backup integrity BEFORE restoring. Verify backup timestamps predate patient-zero's initial access — ransomware operators often compromise backup infrastructure first
    4h
  • 5.2
    Reset ALL credentials — assume full domain compromise
    krbtgt reset (twice, 10h apart for Kerberos ticket expiry) · All domain admin, service account, and privileged user passwords · MFA re-enrollment for all admins · VPN certificates if applicable
    8h
  • 5.3
    Prioritize recovery order by business impact tier
    Tier 1: life-safety, payment, authentication infrastructure · Tier 2: core business operations · Tier 3: productivity/collaboration · Tier 4: non-critical
    variable
  • 5.4
    Validate each recovered system with EDR before returning to production
    Reinstall EDR agent on clean base. Confirm no persistence mechanisms remain (scheduled tasks, registry run keys, WMI subscriptions, services)
    variable

01 Detection & Initial Triage
  • 1.1
    Characterize the detection: how was exfiltration discovered?
    DLP alert · SIEM anomaly (large outbound to new destination) · Threat intel (data on dark web marketplace) · Third-party notification · Attacker ransom demand citing stolen data
    10min
  • 1.2
    Identify the access vector: external intrusion or internal actor?
    Check authentication logs for anomalous access: off-hours, impossible travel, new IP, new user-agent string. Review API access logs and file server audit logs
    20min
  • 1.3
    Engage Legal immediately — clock for regulatory notification starts now
    GDPR: 72h to notify supervisory authority from discovery. HIPAA: 60 days for individuals, 60 days for OCR. State breach laws vary (CA: 72h, NY: 30d). Do not confirm breach publicly until Legal approves
    15min
02 Scope — What Was Accessed?
  • 2.1
    Enumerate all data repositories the compromised account/credential could access
    AD group memberships · SharePoint/OneDrive permissions · S3 bucket policies · Database roles · API OAuth scopes · SaaS app permissions (Salesforce, Workday, etc.)
    1h
  • 2.2
    Quantify exfiltrated data from network logs
    Proxy/firewall logs: bytes transferred per session, destination IPs. CASB logs for cloud uploads. DNS logs for high-volume queries (DNS tunneling). Correlate file access audit logs with data transfer volumes
    2h
  • 2.3
    Classify data types accessed: PII, PHI, financial, IP, credentials?
    Data classification determines regulatory obligations. PII/PHI = GDPR/HIPAA. Payment card data = PCI DSS breach notification. Trade secrets = potential economic espionage. Each triggers different notification and response requirements
    2h
# Reconstruct attacker's access path — proxy logs $ grep "10.1.5.42" /var/log/squid/access.log | awk '{print $1,$7,$9,$10}' | sort -n # S3 CloudTrail — what objects were accessed/downloaded $ aws cloudtrail lookup-events \ --lookup-attributes AttributeKey=EventName,AttributeValue=GetObject \ --start-time 2025-03-01 --end-time 2025-03-08 \ --query 'Events[].{User:Username,Bucket:CloudTrailEvent}' \ | jq '.[].Bucket | fromjson | .requestParameters' # Identify data leaving the network by volume $ tcpdump -r capture.pcap -nn 'dst not 10.0.0.0/8' | awk '{print $5}' | sort | uniq -c | sort -rn | head -20

Supply Chain Compromise Playbook

Covers vendor software verification, fleet-wide triage sequencing, dormancy handling, and sinkholing. Reference cases: SolarWinds SUNBURST, 3CX, XZ Utils backdoor.

HIGH–CRITICAL Complexity: Extreme · Scope: Fleet-wide
RUNBOOK PROGRESS
0%
01 Binary Verification — Confirm Compromise
  • 1.1
    Hash every instance of the flagged software/library across your fleet
    Get-FileHash -Algorithm SHA256 "C:\Program Files\Vendor\*.dll" — compare against vendor's published checksums. Even one mismatch = confirmed compromise candidate
    30min
  • 1.2
    Verify code signing certificate is legitimate AND the file contents match
    Supply chain attackers (SolarWinds, 3CX) used legitimate certificates. A valid signature on tampered code is exactly the scenario. Hash must also match
    15min
  • 1.3
    Static-analyze the suspicious binary in Ghidra before running anything
    Look for: dormancy checks (date/time comparisons), environment checks (domain-joined?), DGA logic, encoded C2 strings, shellcode loading from registry/file
    2h
# Fleet-wide hash verification (PowerShell) PS> $VendorHash = "sha256_from_vendor_advisory_here" PS> Get-ChildItem -Path "C:\Program Files" -Recurse -Filter "SuspectLib.dll" | ForEach-Object { $hash = (Get-FileHash $_.FullName -Algorithm SHA256).Hash [PSCustomObject]@{Path=$_.FullName; Hash=$hash; Match=($hash -eq $VendorHash)} } | Where-Object {-not $_.Match} # Linux: verify against vendor checksum $ sha256sum /usr/lib/libxz.so.5.4.6 | diff - vendor-checksums.txt $ objdump -d /usr/lib/libxz.so.5.4.6 | grep -A5 "call" # look for unexpected calls
02 Dormancy-Aware Triage
Key insight: Supply chain backdoors typically include dormancy periods (SolarWinds: 14 days; XZ Utils: conditional on systemd). A sandbox seeing "nothing" means nothing — static analysis and fleet behavioral data matter more than sandbox detonation.
  • 2.1
    Query fleet: install date vs dormancy window
    If backdoor activates 14 days post-install: hosts installed >14 days ago = active. Hosts installed <14 days = dormant. Contain active tier first; monitor dormant tier
    1h
  • 2.2
    Sinkhole the C2 domain/DGA to enumerate active beaconing hosts
    Work with DNS/NOC team: redirect the DGA TLD or C2 domain to an internal honeypot. Log all connecting IPs — this gives you ground-truth active host list without alerting the adversary
    2h
  • 2.3
    Determine if second stage was delivered to any host before sinkholing
    Hunt: unusual child processes of the vendor software, new registry keys, scheduled tasks, services. Check network logs for resolved C2 before sinkhole was in place
    4h

Business Email Compromise (BEC) Playbook

Covers account takeover detection, wire fraud interception, mail-rule backdoor removal, and tenant hardening. Reference cases: Microsoft 365 OAuth abuse, Octo Tempest / Scattered Spider BEC.

HIGH Wire recall window: <24h
RUNBOOK PROGRESS
0%
TIME CRITICAL: Wire fraud has a recall window of approximately 24–72 hours. Initiate wire recall/hold through your bank's fraud team IMMEDIATELY upon confirmation of a fraudulent transfer — before any forensics.
01 Account Takeover Validation
  • 1.1
    Pull Azure AD / Entra ID sign-in logs for the affected account
    Check: impossible travel (sign-in from US + Europe within minutes) · new device/IP · MFA fatigue attacks (many push notifications) · legacy auth bypass (no MFA required for IMAP/SMTP)
    15min
  • 1.2
    Enumerate malicious mail rules created by attacker
    BEC attackers create inbox rules to hide replies, forward to attacker, delete security alerts. Get-InboxRule -Mailbox victim@company.com — look for rules forwarding to external addresses or moving mail to trash
    10min
  • 1.3
    Check OAuth app consents — attackers register apps to maintain access post-password-reset
    Azure portal: Entra ID → Enterprise Applications → filter "Granted by users". Remove any app the user doesn't recognize. This is how attackers survive password resets and maintain M365 access
    10min
# Exchange Online — dump inbox rules PS> Connect-ExchangeOnline PS> Get-InboxRule -Mailbox "victim@company.com" | Select Name,Enabled,ForwardTo,DeleteMessage,MoveToFolder # Azure AD sign-in logs — impossible travel detection PS> Get-AzureADAuditSignInLogs -Filter "userPrincipalName eq 'victim@company.com'" | Select-Object createdDateTime,ipAddress,location,status,clientAppUsed | Sort-Object createdDateTime | Format-Table # OAuth consents PS> Get-MgUserOAuth2PermissionGrant -UserId victim@company.com | Select-Object ClientId,Scope,ConsentType | Format-Table
02 Wire Fraud Interception
  • 2.1
    Contact bank fraud team IMMEDIATELY with transfer details
    Request SWIFT recall or wire reversal. Provide: date/time, amount, beneficiary bank/account, SWIFT/routing number. US: contact FBI IC3 and FinCEN Financial Fraud Kill Chain (FFKC)
    IMMEDIATE
  • 2.2
    Preserve all email threads related to the fraudulent transfer request
    Legal hold on mailbox. Export headers, body, attachments. Check if email was impersonation (spoofing) vs account takeover — different forensic paths
    30min
  • 2.3
    File FBI IC3 complaint and local law enforcement report
    FBI IC3 (ic3.gov) handles wire fraud. Report within 72h for best chance of fund recovery through the FFKC program. Provide all transaction details and fraud communication evidence
    4h
03 M365 Tenant Hardening
  • 3.1
    Revoke all sessions and tokens for compromised accounts
    Revoke-AzureADUserAllRefreshToken -ObjectId user_id and reset password. Remove all OAuth consents. This invalidates existing access tokens but attacker may re-authenticate if credential still valid
    5min
  • 3.2
    Enable Conditional Access: require MFA, block legacy auth protocols
    Block: IMAP, POP3, SMTP AUTH, Exchange ActiveSync without MFA. These bypass MFA by design and are the primary BEC entry vector. Require compliant device for M365 access
    2h
  • 3.3
    Enable Microsoft Secure Score remediations specific to BEC
    Turn on: DMARC+DKIM+SPF enforcement · Anti-phishing policy (impersonation protection) · Safe Links · Disable auto-forwarding to external domains (anti-exfil rule)
    1h
