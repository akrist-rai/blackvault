Disk & File System Forensics DF

Sleuth Kit (TSK) DF

mmls · fls · icat · istat · blkstat

The standard open-source disk forensics library. Navigate partition tables, list files by inode, extract file contents by address — all without mounting the volume.

  • mmls disk.imgdisplay partition layout (offsets in sectors)
  • fls -o 2048 disk.imglist all files at partition offset 2048
  • fls -r -m / disk.imgrecursive listing for timeline body
  • icat -o 2048 disk.img 14extract file by inode number
  • istat -o 2048 disk.img 14inode metadata (timestamps, blocks)
  • blkstat disk.img 4096block allocation status
Field tip: Always use the sector offset from mmls with every subsequent command — wrong offset = wrong filesystem.
apt: sleuthkit

xxd / hexdump RE DF

hex inspection · magic bytes

The analyst's first look at any unknown file. Identify format by magic bytes before running any tool — ELF (7F 45 4C 46), PE (4D 5A), ZIP (50 4B), PNG (89 50 4E 47).

  • xxd file | head -4first 64 bytes in hex + ASCII
  • xxd -l 512 disk.imgfirst 512 bytes (MBR region)
  • xxd -s 0x100 -l 64 f64 bytes from offset 0x100
  • hexdump -C file | lesspaged hex dump with ASCII sidebar
Field tip: xxd file | grep -i "4d 5a" locates embedded PE headers in a binary blob — useful for carving from memory dumps.
apt: xxd (bundled with vim-common)

photorec / foremost DF

signature-based file carving

Recover files from unallocated space and raw images using header/footer signatures — no file system required. photorec is fast but extracts without original filenames; foremost preserves more metadata.

  • photorec disk.imginteractive carving wizard
  • foremost -i disk.img -o out/carve to output directory
  • scalpel -b -o out/ disk.imgheader/footer carving w/ config
Field tip: Run on a copy or acquired image, never the original. Unallocated space is the target — allocated files are easier to recover via inode.
apt: testdisk (photorec) foremost scalpel

fdisk / gdisk DF

partition table inspection

Quick partition table inspection on disk images or live devices. Use -l for listing only — don't modify a forensic image.

  • fdisk -l disk.imgMBR partition table listing
  • gdisk -l disk.imgGPT partition inspection
  • parted disk.img printformat-agnostic partition view
apt: fdisk gdisk parted
Assembly & Debugging RE

gdb RE

GNU Debugger · single-step · inspect state

The foundation of Linux debugging. Set breakpoints, single-step through instructions, inspect registers and memory. pwndbg and peda plugins improve the default UI significantly.

  • gdb -q ./binaryquiet mode (no banner)
  • break mainset software breakpoint at main
  • hbreak *0xADDRhardware breakpoint (no 0xCC patch)
  • run arg1 arg2start with arguments
  • stepistep one machine instruction (into calls)
  • nextistep over (skip call body)
  • info registersdump all register values
  • x/20i $ripdisplay next 20 instructions
  • x/32xb $rsp32 bytes of stack in hex
  • p/x $raxprint rax in hex
  • disas /r funcdisassemble function with raw bytes
  • dump memory f.bin $s $edump memory range to file
OEP hunting: use hbreak (hardware breakpoint) to catch execution of freshly-written memory — software breakpoints patch 0xCC which anti-tamper code can detect.
apt: gdb · pip: pwndbg

objdump RE

disassembly · section headers · symbols

Quick command-line disassembly and binary inspection. Prefer Intel syntax. Pair with grep to find specific instructions or addresses in large binaries.

  • objdump -d -M intel bindisassemble .text in Intel syntax
  • objdump -j .text -d binonly the .text section
  • objdump -h binsection headers + sizes
  • objdump -t binsymbol table
  • objdump -R bindynamic relocation entries
apt: binutils

readelf RE

ELF header · segments · dynamic symbols

Comprehensive ELF inspection — header, program headers (segments), section headers, dynamic section, symbol tables. The primary tool for understanding ELF structure before running Ghidra.

  • readelf -h binELF header (arch, entry, type)
  • readelf -l binprogram headers (segments)
  • readelf -S binsection headers
  • readelf -d bindynamic section (NEEDED libraries)
  • readelf -s binsymbol tables
  • readelf -a bineverything (very verbose)
apt: binutils

ndisasm RE

raw byte disassembly — no ELF wrapper needed

Disassemble raw bytes (shellcode, firmware blobs) without ELF/PE headers. Essential for analyzing memory-dumped shellcode or extracted payloads.

  • ndisasm -b64 shellcode.bin64-bit flat disassembly
  • ndisasm -b32 -o 0x400000 f32-bit at custom base address
Inline disassembly: pipe hex from xxd or use echo -ne '\x48\x31\xc0' | ndisasm -b64 - to disassemble individual instruction bytes.
apt: nasm
Decompilers & Binary Analysis RE

Ghidra RE

NSA · free · full decompiler + type recovery

The most capable free RE platform. Import a binary, run auto-analysis, then use the decompiler + listing view to reverse engineer functions. Define structs and prototypes to improve pseudocode quality iteratively.

  • ghidraRunlaunch GUI (GHIDRA_HOME/support/)
  • analyzeHeadless proj X -import binnon-GUI scripted analysis
  • analyzeHeadless proj X -postScript script.pyrun a post-analysis script
Workflow: (1) right-click function → Edit Function Signature. (2) retype parameters. (3) re-decompile — the pseudocode updates everywhere. (4) use xrefs (Ctrl+Shift+F) to jump to all callers. Repeat until logic reads clearly.
Download: github.com/NationalSecurityAgency/ghidra

radare2 (r2) RE

command-line RE powerhouse · scriptable · patcher

Full RE suite in the terminal: disassembly, analysis, pseudo-decompilation (pdc), binary patching, and scripting via r2pipe. Steep learning curve; extremely powerful for automation and quick triage.

  • r2 -A binaryopen + full auto-analysis
  • s mainseek to main function
  • pdf @ mainprint disassembly of function
  • pdcpseudo-decompiler output
  • afllist all analyzed functions
  • izlist strings in binary
  • iiimports (IAT/GOT)
  • r2 -w binopen in write mode (for patching)
  • wa nopwrite NOP at current seek position
  • px 32 @ 0x401000print 32 hex bytes at address
apt: radare2 · or: r2env from github

rabin2 RE

binary metadata extraction (part of radare2)

Quick binary identity and structure extraction without opening a full RE session. Part of the radare2 suite but usable standalone for rapid triage.

  • rabin2 -I binarybinary identity (arch, bits, type, OS)
  • rabin2 -i binaryimports (functions binary calls)
  • rabin2 -e binaryexports / symbols
  • rabin2 -S binarysections with sizes and entropy
  • rabin2 -z binarystrings in data sections
  • rabin2 -H binaryheader info (PE or ELF)
apt: radare2
Static Malware Analysis MA

strings MA

printable-string extraction — always first

The cheapest, fastest static signal. Always run before anything else. URLs, IPs, registry paths, mutex names, error messages, and hardcoded passwords all appear here — until the sample is packed.

  • strings -a sampleASCII strings (entire file)
  • strings -a -el sample16-bit little-endian Unicode too
  • strings -n 8 sampleminimum length 8 (fewer noise hits)
  • strings sample | grep -Ei 'http|cmd|temp|run'filter for high-value patterns
Empty output? The binary is packed or obfuscated. Escalate to FLOSS, then dynamic analysis. Absence of strings is itself a packing indicator.
apt: binutils

FLOSS MA

FLARE Obfuscated String Solver · recovers hidden strings

Finds strings that strings misses: stack-built strings, decoded strings, and encoding routines. The critical next step when plain strings returns nothing useful.

  • floss sample.exeextract all string types
  • floss --no-static sample.exeskip plain strings, focus on obfuscated
  • floss -j out.json sample.exeJSON output for automation
pip: flare-floss · or: github.com/mandiant/flare-floss

YARA MA

pattern-matching rule engine for malware signatures

Write rules that match on strings, byte patterns, and boolean conditions. The analyst's signature workhorse for detection, hunting, and family classification. Rules compose with logical operators for precision.

  • yara rules.yar samplerun one ruleset against one file
  • yara -r rules.yar /samples/recursive scan of directory
  • yara -s rules.yar sampleshow matching strings
  • yarac rules.yar rules.yarccompile for faster scanning
Rule structure: rule name { strings: $s = "malware" condition: $s } — start simple and layer: add byte patterns, PE magic, and entropy conditions.
apt: yara · pip: yara-python

capa MA

FLARE Labs · capability detection mapped to ATT&CK

Infers named capabilities from code and API patterns, then maps them to MITRE ATT&CK and MBC without executing the sample. Produces a behavioral preview from static analysis alone.

  • capa sample.exefull capability report
  • capa -j sample.exeJSON output for automation
  • capa -v sample.exeverbose — shows matching addresses
pip: flare-capa · or github releases

ssdeep / imphash MA

fuzzy + import hashing for sample clustering

Cluster related samples without exact hash matching. ssdeep (CTPH) survives minor byte changes; imphash fingerprints the import table to identify same-toolchain builds even after repacking.

  • ssdeep -b sample.execompute fuzzy hash
  • ssdeep -m known.ssdeep sample.execompare against reference set
  • python3 -c "import pefile; pe=pefile.PE('s.exe'); print(pe.get_imphash())"compute imphash
apt: ssdeep · pip: pefile
Dynamic Analysis & Instrumentation MA

strace MA

system call tracer — the un-lieable boundary

Every file open, network connection, process spawn, and persistence write crosses the kernel boundary as a syscall. Obfuscation hides code, not syscalls — strace is the ultimate truth at that boundary.

  • strace ./sampletrace all syscalls
  • strace -e trace=file ./samplefile-related syscalls only
  • strace -e trace=network ./samplenetwork syscalls only
  • strace -f ./samplefollow forked/exec'd children
  • strace -o run.log -f ./samplewrite to file (often large)
  • strace -p PIDattach to running process
Statically-linked sample: ltrace is blind (no PLT); strace still sees all syscalls. Always prefer strace as your primary dynamic tool.
apt: strace

ltrace MA

library call tracer — hooks PLT/GOT stubs

Traces calls to dynamic library functions (libc, libssl, etc.) and their arguments. Useful for dynamically-linked samples; blind for statically-linked ones (no PLT to hook).

  • ltrace ./sampletrace all library calls
  • ltrace -e strcmp ./sampletrace only strcmp calls
  • ltrace -l libssl.so ./sampletrace a specific library
apt: ltrace

Frida MA RE

dynamic instrumentation — hook any function at runtime

Inject JavaScript into a running process to hook functions, read/write memory, and observe arguments. The key tool for hooking the encrypt-before-send routine in Protocol RE (Phase 10) to recover keys and plaintext without breaking the cipher.

  • frida -l hook.js -f ./samplespawn process and load script
  • frida -l hook.js -p PIDattach to running process
  • frida-trace -i "send" -p PIDauto-hook send() with arg logging
  • frida-ps -alist running apps + PIDs
Hook template: Interceptor.attach(ptr('0xADDR'), { onEnter(args) { console.log(args[0].readByteArray(32)); } }); — adjust the address from your Ghidra/radare2 analysis.
pip: frida frida-tools

inotifywait MA

live filesystem event monitoring

Watch exactly what files a sample creates, modifies, or deletes as it runs. Set up a bait directory and monitor it during detonation — no before/after snapshot diff needed.

  • inotifywait -mr ~/bait -e modify,create,deleterecursive monitoring of bait dir
  • inotifywait -m /etc /tmp -e createwatch persistence/staging paths
apt: inotify-tools
Unpacking & Packing Detection MA

UPX MA

Ultimate Packer for eXecutables · common on malware

The most common malware packer. Fortunately, trivially reversible with upx -d. Recognizable by UPX0/UPX1 section names and near-8.0 entropy. Many families strip section names but keep the structure.

  • upx -d packed.exe -o out.exeunpack (if headers intact)
  • upx --info packed.exedetect and show packer info
  • upx -l packed.exelist compressed sections
Stripped headers? Many samples overwrite "UPX0" in the section names. The structure remains the same — unpack manually by finding the tail jump and dumping at OEP.
apt: upx-ucl

ent MA

entropy measurement — packing/encryption indicator

Measure bits-per-byte entropy of a file or section. Near 8.0 = compressed or encrypted. Combine with a small import table to flag packing. High entropy alone is not proof of malice.

  • ent sample.exefull file entropy (bits/byte)
  • dd if=sample.exe bs=512 count=1 | ententropy of first 512 bytes
Baseline: a typical unpacked ELF text section is ~5.5–6.5 bits/byte. A packed section is 7.8–8.0. A random-looking section in an otherwise normal binary is suspicious.
apt: ent
Memory Forensics DF

Volatility 3 DF

memory image analysis framework

The standard memory forensics framework. Walk EPROCESS lists, find injected shellcode via VAD analysis, recover network connections from socket structures, dump process images, and extract credentials — all from a frozen memory image.

  • vol -f mem.raw windows.infoidentify OS / build (get symbols first)
  • vol -f mem.raw windows.pslistactive process list (EPROCESS links)
  • vol -f mem.raw windows.psscanpool-scan — catches DKOM-hidden processes
  • vol -f mem.raw windows.pstreeprocess tree with parent PIDs
  • vol -f mem.raw windows.malfindprivate RX/RWX regions (injected code)
  • vol -f mem.raw windows.netscanTCP/UDP connections + owning PIDs
  • vol -f mem.raw windows.cmdlinefull command lines per process
  • vol -f mem.raw windows.dlllist --pid 1234loaded modules for a process
  • vol -f mem.raw windows.hashdumpextract NT password hashes
  • vol -f mem.raw windows.procdump --pid 1234 --dump-dir out/dump process image to disk
Cross-view: Always run both pslist AND psscan. Anything in psscan but not pslist = DKOM-hidden process. This single check catches most kernel-level stealth.
pip: volatility3 · or: github.com/volatilityfoundation/volatility3
Network Forensics DF

tcpdump DF

packet capture · command-line · first-look analysis

The command-line packet capture and inspection tool. Capture to pcap files for later Wireshark analysis, or apply BPF filters to inspect traffic in real time. Fast, scriptable, runs on any Linux server.

  • tcpdump -i eth0 -w capture.pcapcapture all traffic to file
  • tcpdump -nr capture.pcapread a pcap (no DNS lookup, no promiscuous)
  • tcpdump -nr cap.pcap 'tcp and dst port 443'BPF filter: HTTPS only
  • tcpdump -nr cap.pcap 'host 185.220.101.47'filter by host
  • tcpdump -nr cap.pcap -c 20first 20 packets only
apt: tcpdump

tshark / Wireshark DF

pcap analysis · protocol dissection · file carving

Full protocol-aware packet inspection. Start with Statistics → Conversations to get a top-down view, then drill into individual streams. Export Objects carves transferred files (PE binaries, scripts). Never scroll packet-by-packet for a large capture.

  • tshark -r cap.pcap -q -z conv,ipIP conversation summary
  • tshark -r cap.pcap -Y "http.request"filter for HTTP requests
  • tshark -r cap.pcap -Y "dns" -T fields -e dns.qry.nameextract all DNS queries
  • tshark -r cap.pcap --export-objects http,out/carve HTTP transferred files
  • tshark -r cap.pcap -z io,stat,60traffic volume per 60-second interval
apt: tshark wireshark

Zeek (Bro) DF

structured network logs — triage at scale

Zeek turns a pcap into typed logs (conn.log, dns.log, ssl.log, http.log, files.log) that you query like a database. For large captures, Zeek is how you answer "which internal hosts talked to new domains" without scrolling 2 GB of packets.

  • zeek -r capture.pcapgenerate all log files in current dir
  • cat conn.log | zeek-cut id.orig_h id.resp_hpull source/dest pairs
  • cat dns.log | zeek-cut query answersDNS queries + responses
  • cat ssl.log | zeek-cut ja3 server_nameJA3 + SNI per TLS session
apt: zeek · or: zeek.org

Suricata DF

IDS/IPS · run signature rules over a pcap

Run community or custom rules against a recorded pcap to rapidly surface known-bad IOCs. Works offline — point it at a pcap with a ruleset and get alerts in seconds.

  • suricata -r capture.pcap -S rules.rules -l logs/run against pcap, log to dir
  • suricata-updateupdate Emerging Threats rulesets
  • cat logs/fast.logquick alert summary
apt: suricata
Hash Cracking & Cryptanalysis DF

hashcat DF

GPU-accelerated hash cracking

The fastest CPU/GPU hash cracker. Supports 300+ hash modes. Use dictionary attacks first (common passwords), then rule-based mutations, then brute force for short hashes.

  • hashcat -m 0 hash.txt wordlist.txtMD5 dictionary attack
  • hashcat -m 1000 hash.txt wordlist.txtNTLM dictionary attack
  • hashcat -m 0 hash.txt -a 3 ?l?l?l?lMD5 brute force (4 lowercase)
  • hashcat -m 0 hash.txt wordlist.txt -r rules/best64.ruledictionary + mutations
apt: hashcat

John the Ripper DF

CPU-based password / hash cracker

Supports a wide range of hash and file formats (zip, PDF, RAR, KeePass, SSH keys). The --format flag is critical — wrong format = no cracks.

  • john --format=md5crypt hash.txtcrack with auto wordlist
  • john hash.txt --wordlist=words.txtdictionary attack
  • john hash.txt --showdisplay cracked passwords
  • zip2john archive.zip > hash.txtextract ZIP hash for cracking
apt: john
Windows Artifact Parsers (Eric Zimmerman Tools) DF

MFTECmd DF

NTFS $MFT parser — timestamps + SI/FN comparison

Parse the NTFS $MFT to a CSV timeline. Crucially, exports both $STANDARD_INFORMATION and $FILE_NAME timestamps so you can detect timestomping — where $SI is edited but $FN (kernel-maintained) still shows the real time.

  • MFTECmd -f '$MFT' --csv out/parse to CSV timeline
  • MFTECmd -f '$MFT' --csv out/ --csvf mft.csvnamed output file
Timestomping tell: $SI Created newer than $FN Created, and $SI timestamps are suspiciously round numbers (00:00:00 exactly). The kernel sets $FN; user-mode tools can only set $SI.
Download: ericzimmerman.github.io/#!index.md

EvtxECmd DF

Windows Event Log (EVTX) parser

Parse EVTX files to structured CSV/JSON for timeline analysis. Key Event IDs: 4624 (logon), 4688 (process creation), 7045 (service install), 1102 (log cleared), 4104 (PowerShell scriptblock).

  • EvtxECmd -d C:\evtx --csv out/parse all logs in directory
  • EvtxECmd -f Security.evtx --csv out/one log file
Download: ericzimmerman.github.io

AppCompatCacheParser / AmcacheParser DF

execution evidence surviving file deletion

ShimCache (AppCompatCache) and AmCache record binary presence and execution evidence even after the file is deleted. Critical for proving a malicious binary ran when the attacker has cleaned up.

  • AppCompatCacheParser -f SYSTEM --csv out/parse ShimCache from SYSTEM hive
  • AmcacheParser -f Amcache.hve --csv out/parse AmCache
Download: ericzimmerman.github.io

plaso / log2timeline DF

super-timeline — merge all artifact sources

Ingest a disk image (or directory of artifacts) and produce a single merged timeline spanning MFT, event logs, registry, browser history, shellbags, prefetch, and more. The capstone Phase 12 tool.

  • log2timeline.py out.plaso image.ddingest image → timeline store
  • psort.py -o l2tcsv out.plaso -w timeline.csvfilter / export to CSV
  • psort.py out.plaso "timestamp > '2026-06-01'"filter by date range
pip: plaso · or: github.com/log2timeline/plaso
